Sprint 3 introduces the full backend infrastructure: a FastAPI Python service, PostgreSQL database, Redis cache, Alembic migrations, and a seed script generating 100 products and 5,000 reviews. ## Running the Backend

### With Docker (recommended)
```bash
docker-compose up --build
```
This starts:
- **FastAPI** on http://localhost:8000
- **PostgreSQL** on localhost:5432
- **Redis** on localhost:6379

### Seed the database
```bash
docker-compose exec api python -m app.seed.seed_data
```

### Run migrations manually
```bash
cd backend
alembic upgrade head
```

### API docs (auto-generated by FastAPI)
```
http://localhost:8000/docs
```

## Architecture: SQLAlchemy Async

Uses SQLAlchemy 2.0 with async engine (`asyncpg` driver):

```python
engine = create_async_engine(settings.database_url)
AsyncSessionLocal = async_sessionmaker(engine, expire_on_commit=False)

# In every router — injected via FastAPI Depends:
async def get_db() -> AsyncSession:
    async with AsyncSessionLocal() as session:
        yield session
```

All queries use `await db.execute(select(...))` — never blocking the event loop.

## Redis Caching Pattern

Every router caches its response in Redis. Cache failures are **silent** — the API continues working without Redis:

```python
async def cache_get(key: str) -> dict | None:
    try:
        raw = await get_redis().get(key)
        return json.loads(raw) if raw else None
    except Exception as e:
        logger.warning("Cache GET failed for %s: %s", key, e)
        return None  # ← fallback to DB
```

Cache TTL defaults to 300s (5 min). Search results use a shorter 120s TTL.

## Frontend: Dual-Mode Hooks

`src/hooks/useProducts.ts` provides 6 hooks that work in two modes:

| Hook | What it returns |
|------|----------------|
| `useFeaturedProduct()` | Hero product for homepage |
| `useTopReviewed(category?)` | Top-rated products grid |
| `useProducts(params)` | Filtered/sorted product list |
| `useProduct(id)` | Single product + pros/cons/highlights |
| `useReviews(id, filter)` | Reviews with optional filter |
| `useSearch(query)` | Search results |

**Mode detection:**
```ts
export const API_ENABLED = Boolean(import.meta.env.VITE_API_URL)
```

When `VITE_API_URL` is NOT set (default in dev/tests):
- Mock data is returned **synchronously** via `useMemo`
- No loading state, no async overhead
- All 39 existing tests pass without `waitFor`

When `VITE_API_URL` IS set (e.g., `.env.local`):
- Hooks fetch from the real API
- `loading: true` on first render
- `error` contains message if fetch fails

### Enabling the API
```env
# .env.local
VITE_API_URL=http://localhost:8000
```

## Adapter: ApiProduct → Product

The API returns a different shape than the mock data (e.g., `review_count` vs `reviewCount`, `price: number` vs `priceRange: string`). The `adaptProduct()` function normalises this so pages work identically in both modes:

```ts
function adaptProduct(p: ApiProduct, rank = 0): Product {
  return {
    ...p,
    priceRange: `$${p.price.toLocaleString()}`,
    reviewCount: p.review_count,
    rank,
    // gradient colours derived from category
    ...CATEGORY_COLORS[p.category],
  }
}
```
